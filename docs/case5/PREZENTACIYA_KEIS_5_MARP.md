# Юзабилити-тестирование
## Продукт: «Заметки мероприятий» (Flutter)
### Кейс-задача № 5 · май 2026

---

## Введение

**Что тестировали:** мобильное / десктопное приложение для **заметок на совещаниях и деловых встречах** — создание, поиск, редактирование, удаление; хранение **SQLite** или **файлы на устройстве**.

**Зачем:** оценить, насколько интерфейс **поддерживает реальную работу** организации без длительного обучения и где возникают **риски ошибок и потери доверия** к инструменту.

**Для кого этот доклад:** научный руководитель и заказчик внедрения — акцент на **последствиях для процессов** и **выгодах** от приоритетных доработок.

---

## Цели тестирования

| Направление | Что измеряли |
|-------------|----------------|
| **Эффективность** | Время на типовые задачи без пошаговой инструкции |
| **Ошибки** | Случаи, ведущие к **неверным действиям** или **ощущению потери данных** |
| **Удовлетворённость** | Субъективная «лёгкость» после сценария (шкала 1–7) |
| **Обучаемость** | Нужен ли отдельный онбординг для нетехнической аудитории |

---

## Методы и подходы

- **Формат:** модерируемые сессии **1 : 1**, **think-aloud** (думаю вслух).
- **Площадка:** MacBook (`flutter run -d macos`) и/или личный смартфон участника.
- **Протокол:** время шагов, ошибки, **короткие цитаты**, без агрессивных подсказок.
- **Этика:** информированное согласие; данные в отчёте — **обезличенно** (P1…P4).

**Ограничение:** выборка **N = 4** — типично для **формативных** UX-исследований (поиск паттернов, не рыночная статистика).

## Участники (целевая аудитория)

Смешанный **организационный** профиль, **не** только IT:

| Участник | Роль / контекст |
|----------|-----------------|
| **P1** | HR, служебные встречи и протоколы |
| **P2** | Руководитель проекта, планёрки |
| **P3** | Стажёр, низкая привычка к кастомным приложениям |
| **P4** | Бухгалтерия / отчётность, осторожный пользователь |

---

## Сценарий: задачи T1–T4 (реалистичные формулировки)

1. **T1** — Создать заметку на **завтра 10:00** о **совещании по бюджету** с кратким заголовком и текстом (роль, вопрос повестки).
2. **T2** — **Найти** заметку по ключевому слову (например, «подрядчик») **через поиск**.
3. **T3** — **Сдвинуть время** встречи на **+1 час**, сохранить.
4. **T4** — **Удалить** заранее помеченный черновик.

*Участнику не говорили названия кнопок — только бизнес-задачи.*

---

## Сценарий: задача T5 (хранилища)

**T5:** переключить **тип хранения** (SQLite ↔ файлы) и объяснить, **куда делись / почему видны** старые заметки.

**Цель:** выявить риск **ошибочной модели** данных при внедрении в организации.

## Результаты: качественные находки

- **Поиск:** часть пользователей сначала **скроллит список**; вопросы к **формату даты** в запросе.
- **Два хранилища:** после переключения — **неожиданность**, что список «не тот»; риск объяснения «данные пропали».
- **Удаление:** диалог подтверждения воспринят **положительно**; на macOS ждали **контекстное меню**.

## Ключевые проблемы → бизнес-последствия

| Проблема | Последствия |
|----------|-------------|
| Непонятные **следствия смены хранилища** | Паника, обращения в поддержку, **потеря доверия** к инструменту |
| **Поиск** и формат даты | Лишнее время на каждой итерации → **накопительный** эффект по организации |
| Нет **онбординга** | Зависимость от внутренних тренингов при внедрении |

---

## Рекомендации (приоритеты)

| Приоритет | Действие |
|-----------|----------|
| **P0** | Модальное предупреждение при **смене хранилища** + явный текст про **независимые** наборы заметок |
| **P0** | Короткий **онбординг** при первом запуске (создание → поиск → где хранилище) |
| **P1** | Упростить подсказку **поиска** и пример допустимого **формата даты** |
| **P1** | Вынести **«Хранилище»** в понятный раздел «Настройки» |
| **P2** | Контекстное меню на десктопе; в перспективе — **экспорт / резервная копия** |

---

## Потенциальная выгода для организации

- **Меньше инцидентов** «пропали заметки» → ниже **репутационный** и **операционный** риск внедрения.
- **Быстрее поиск** контекста после встречи → экономия **рабочего времени** множества сотрудников.
- **Меньше часов обучения** — экономия **ФОТ** на внутренних тренингах при масштабировании.

*Количественно связать с KPI (часы/год) можно на этапе пилота.*

## Заключение

Тестирование подтвердило **жизнеспособность** основного сценария (создать — найти — изменить — удалить), но выявило **управляемые** риски вокруг **модели данных** и **поиска**.

Предложенные доработки **P0** снижают барьер **доверия** — критично для **внедрения в организации**. Следующий шаг — **повторное** юзабилити после правок (итерация).
