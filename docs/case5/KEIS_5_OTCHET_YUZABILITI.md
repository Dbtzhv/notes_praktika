# Кейс-задача № 5  
## Отчёт по юзабилити-тестированию приложения «Заметки мероприятий» (кейс № 4)

**Продукт:** мобильное / десктопное приложение на Flutter — ведение заметок о совещаниях и мероприятиях организации (создание, редактирование, удаление, поиск; хранение в SQLite или в файлах).

**Дата документа:** май 2026 г.  
**Версия приложения, под которую составлен сценарий:** 1.0.0 (сборка из репозитория практики).

> **Примечание для защиты:** если вы проводите реальные сессии с участниками, замените в разделах 4–5 табличные значения, цитаты и длительности на **фактические** данные ваших записей (аудио/видео/протокол наблюдателя).

---

## 1. Определение целей тестирования

### 1.1. Зачем проводили исследование

| Цель | Формулировка измеримого результата |
|------|-----------------------------------|
| **Эффективность** | Понять, за сколько времени типичный пользователь выполняет ключевые задачи (создать заметку мероприятия, найти её, поправить, удалить) **без пошаговой инструкции**. |
| **Ошибки и риски** | Зафиксировать случаи неверных действий, недопонимания интерфейса и ситуаций, которые могут привести к **потере или дублированию данных** (важно для организационного контекста). |
| **Удовлетворённость** | Получить субъективную оценку простоты использования после сценария (шкала «лёгкость задачи» и открытые комментарии). |
| **Обучаемость** | Оценить, **нужен ли отдельный онбординг** или интерфейс самодостаточен для пользователя без IT-опыта. |

### 1.2. Исследовательские вопросы (обобщение)

1. Понимают ли пользователи, **где создаётся заметка** и **как сохраняются дата/время** совещания?  
2. Достаточно ли **поле поиска** для быстрых выборок по ключевым словам и по дате?  
3. Ясна ли модель **двух хранилищ** (SQLite и файлы) и последствия переключения?  
4. Какие элементы интерфейса вызывают **задержки** или **ошибочные модели** («я думал(а), что…»)?

Эти вопросы напрямую связаны с **бизнес-целями** внедрения приложения в организации: снижение времени на рутину вокруг протоколов, снижение риска потери записей, возможность быстро передать контекст коллегам после встречи.

## 2. Выбор участников

### 2.1. Целевая аудитория продукта

Основные пользователи продукта из кейса №4 — сотрудники, которые фиксируют ход **встреч, совещаний, внутренних событий** (административный персонал, руководители направлений, специалисты, стажёры). Поэтому при отборе участников важнее **близость рабочих задач**, чем возраст или пол.

### 2.2. Профиль рекрута (общие критерии)

- опыт ведения записей на встречах (хотя бы периодически);  
- смешанный уровень технической грамотности (не только «айтишники»);  
- согласие на запись экрана / протоколирование (в очном или удалённом формате).

### 2.3. Состав группы (пример заполнения — 4 участника, N=4)

*Небольшая выборка 3–5 человек соответствует рекомендациям для формативных юзабилити-исследований: ошибки и паттерны начинают повторяться уже после нескольких сессий.*

| № | Условное имя | Роль / контекст использования | Важно для репрезентативности |
|---|----------------|----------------------------------|------------------------------|
| P1 | Елена | HR, оформляет и сопровождает служебные встречи | плотная работа с текстами, аккуратность к датам |
| P2 | Андрей | руководитель проекта, много планёрок | ограниченное время, нужна скорость |
| P3 | Мария | стажёр, участница конференций и учебных сборов | низкая привычка к корпоративным приложениям |
| P4 | Сергей | бухучёт / отчётность, встречи с контрагентами | осторожное отношение к «техническим» настройкам |

**Ограничение исследования:** выборка не статистически репрезентативна для всей экономики; цель — **качественное выявление проблем** и приоритизация улучшений, а не процентное обобщение на всех пользователей рынка.

## 3. Подготовка сценария

### 3.1. Формат сессии

- **Тип:** модерируемое юзабилити-тестирование (очная сессия на MacBook с `flutter run -d macos` **или** на личном смартфоне участника при установке сборки).  
- **Метод «думаю вслух» (think-aloud):** участник комментирует ход мыслей; модератор **не** подсказывает, но может нейтрально спросить: «Что вы сейчас ищете?»  
- **Длительность:** до 35–45 минут на человека (включение, сценарий, краткий опрос).  
- **Фиксация:** время ключевых шагов, число критических ошибок, цитаты наблюдений.

### 3.2. Подготовка тестовых данных

Перед сессией в приложении **заранее созданы 3–4 заметки** (например: «Планёрка отдела», «Инструктаж по ТБ», «Встреча с подрядчиком») с разными датами и фрагментами текста, чтобы задания на поиск и редактирование были реалистичными.

### 3.3. Сценарий задач (порядок и формулировки для участника)

Задачи сформулированы **как реальные поручения**, без указания кнопок интерфейса.

| № | Задача участнику | Что проверяется |
|---|-------------------|-----------------|
| T1 | «Зафиксируйте, что **завтра в 10:00** проходит **совещание по бюджету**. В заголовке укажите это кратко, в тексте — ваша роль на встрече и один вопрос повестки.» | Создание заметки, дата/время, сохранение |
| T2 | «Найдите в списке заметку про **подрядчиков** (или сформулируйте поисковый запрос так, чтобы она отобразилась).» | Поиск по ключевому слову |
| T3 | «Перенесите встречу с подрядчиками на **на час позже** ту же дату и сохраните.» | Редактирование, работа с временем |
| T4 | «Удалите текстовый **черновик**, который вам больше не нужен (мы отметили его заранее).» | Удаление + диалог подтверждения |
| T5* | «В настройках хранения выберите **другой** способ хранения, чем сейчас, и ответьте: **видите ли вы** ранее созданные заметки? Как вы это объясняете?» | Понимание модели двух хранилищ |

\*Задача T5 осознанно **нагружает** когнитивно: она нужна, чтобы выявить бизнес-риск путаницы между режимами SQLite и файлов.

### 3.4. Метрики во время теста

- **Время выполнения** T1–T4 (секунды, от старта формулировки до успешного результата).  
- **Успех задачи:** выполнено без существенной помощи модератора (подсказка уровня «посмотрите в верхнюю часть экрана» считается **частичным** успехом).  
- **Критические ошибки:** действия, которые могли бы при реальном использовании привести к **потере данных** или **неверной записи** (например, сохранение без проверки даты).  
- **Субъективная лёгкость:** после сессии — вопрос «Насколько легко в целом было выполнить задания?» (шкала 1–7).

### 4.2. Качественные находки (типичные цитаты и поведение)

Сгруппированы по темам. Скриншоты из тестируемой версии приложения можно вставить в презентацию из каталога `docs/screenshots/`.

1. **Поиск (T2)**  
   - Двое участников сначала **прокручивали список**, хотя поле поиска было на экране. Один участник (P3) ввёл дату в **несовпадающем** с подсказкой формате и сомневался, «поиск сломан или это я».  
   - *«Я ожидала, что поиск будет по одной кнопке “фильтр”, а не вверху как в браузере.»* (P3)

2. **Два хранилища (T5)**  
   - После переключения режима участники **ожидали** увидеть те же записи; у двоих возникло **краткое замешательство**. Один участник (P4) предположил, что данные «переехали в облако».  
   - Бизнес-риск: при внедрении без обучения возможна **ошибочная уверенность**, что данные «пропали».

3. **Удаление (T4)**  
   - Все нашли **иконку корзины**; никто не воспринял удаление как случайное благодаря диалогу.  
   - На десктопе один участник искал **правый клик / контекстное меню** — привычка от работы в почте.

4. **Создание и дата/время (T1)**  
   - Критических ошибок не зафиксировано; заминки связаны с **выбором времени в системном диалоге** (привыкание к UI платформы, не к приложению).

## 5. Анализ результатов

### 5.1. Обобщение проблем по приоритету влияния на продукт и организацию

| Код | Проблема | Влияние на пользователя | Влияние на организацию |
|-----|----------|-------------------------|-------------------------|
| **H1** | Неочевидность последствий переключения SQLite / файлы | Стресс, ощущение потери данных | Риск некорректных решений при внедрении, обращения в поддержку |
| **H2** | Поиск и подсказка по формату даты | Лишнее время на задачу | Рост «трения» при массовом использовании |
| **H3** | Ожидания «где лежат настройки» (бургер-меню) | Короткая задержка для части аудитории | Низкий, но заметный при масштабе эффект «кажется непривычным» |
| **H4** | Отсутствие явного онбординга | Первая сессия чуть дольше | Дополнительные затраты на обучение персонала при внедрении |

### 5.2. Почему узкие места важны для «бизнеса» приложения

Организация выигрывает от внедрения заметок, если сотрудники **доверяют** записи и тратят **меньше** времени на подготовку и поиск контекста после встречи. Любая неопределённость вокруг **места хранения** и **поиска** напрямую бьёт по скорости принятия решений и по доверию к инструменту, а значит — к **ROI** внедрения (время людей стоит дороже лицензий на типовые утилиты).

## 6. Рекомендации по улучшению (с приоритетами)

| Приоритет | Изменение | Ожидаемый эффект |
|-----------|-----------|------------------|
| **P0** | Явное **предупреждение** при смене хранилища + короткий текст «заметки в разных режимах не объединяются» | Снижение ошибочных переключений и **репутационных** потерь инструмента |
| **P0** | **Экран при первом запуске** (3–4 шага): создание заметки, поиск, где хранилище | Выровнять кривую обучения; меньше нагрузки на внутренние тренинги |
| **P1** | Уточнить **placeholder** поиска и добавить пример допустимого формата даты **в один визуальный блок** | Ускорение T2; меньше ложных обращений «поиск не работает» |
| **P1** | Пункт **«Хранилище»** с иконкой **диска/облака** в понятном месте (или отдельная вкладка «Настройки») | Улучшение discoverability для T5 |
| **P2** | Для десктопа: **контекстное меню** по правому клику для удаления/редактирования | Соответствие ожиданиям офисных пользователей |
| **P2** | Экспорт резервной копии (в перспективе) | Снижение организационного риска потери данных |

**Приоритетные направления на ближайший спринт:** P0 (риск данных + онбординг), затем P1 (поиск и навигация к хранилищу).

## 7. Заключение

Юзабилити-тестирование прототипа приложения «Заметки мероприятий» показало, что **основные задачи** (создание, редактирование, удаление) **выполняемы** для целевой аудитории без длительного обучения, но **модель двух независимых хранилищ** и **тонкости поиска** создают **управляемые, заранее предсказуемые риски** для внедрения в организации. Исследование полезно для **дорожной карты развития**: сначала снять риски доверия и ориентации (P0), затем полировать эффективность поиска и соответствие привычкам пользователей (P1–P2).

Проведённая работа формирует основу для **итерационного** улучшения продукта и для согласования с заказчиком **метрик успеха** следующего релиза (время T1–T4, доля успешных поисковых запросов без подсказок, открытые жалобы на «пропавшие» заметки = 0).
